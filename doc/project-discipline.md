## 優先度の価値観

- リファクタリング vs 新機能: リファクタリング優先
- tech-debt の姿勢: 積極的に返す
- バグ修正: 重篤度・影響範囲による（軽微なものは後回し可）
- 安定性 vs 速度: バランス（MVPを最短距離で）
- 設計原則: YAGNI / KISS / DRY を基本としつつ SOLID を意識する
- 拡張性: 見えている将来の拡張（例: GUI対応）に対して閉じた設計はNG。
  ただし「あるかもしれない拡張」は対象外（YAGNIとの両立）

## 優先度の判断基準

### 工数 vs 優先度
- **低工数 (SP 1–2) の tech-debt は即着手する**。後回しにするコスト（認知負荷の蓄積）が
  工数を上回るため、他 Issue の優先度に関わらず high 扱いにする。
- 工数が大きい (SP 5+) tech-debt は他 Issue との依存・影響範囲を考慮して medium 以下で調整する。

### 依存関係 vs 優先度
- **他 Issue の前提となる Issue（ブロッカー）は、自身の工数・種別に関わらず優先度を上げる**。
  例: SP1 でも他の medium Issue を unlock するなら high にする。

### エラーハンドリング vs 優先度
- **ゲームが止まる / リプレイが壊れる**実害のあるエラー処理抜けは **medium**。
- **サイレントに伝播するが通常は発生しない**エラー処理抜けは **low**。
  放置しても負債が蓄積しないため、他の tech-debt より後回しでよい。
- **low のエラーハンドリング修正には必ず単体テストをセットで実装する**。
  実運用での再現が難しいパスであるため、テストなしの修正は検証不可能になる。
