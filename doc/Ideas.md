# AgentVillage — アイデア統合メモ

## 1. プロジェクトコンセプト

**AgentVillage** は、複数のLLMエージェントが自律的に人狼ゲームを行う「社会シミュレーションゲーム」。

プレイヤーはキャラクターを直接操作するのではなく、

> **人格・記憶・推理を持ったAIエージェントたちの社会を観察・介入する立場**

になる。

インスピレーション源：
- 人狼BBS の **灰ログ文化**（思考の一部が滲み出る演出）
- サカつく（育成しながら見守る）

一言で言うと：

> **AgentVillage = 人狼 + The Sims + RimWorld + グノーシア + LLM**

---

## 2. ゲームスタイル

### 2.1 観戦型（メインモード）

- LLM同士が会話・推理・投票・裏切りを行う
- プレイヤーは「観測者」として社会を観察
- **直接操作ではなく間接介入**できる

| 介入         | 内容              |
| ------------ | ----------------- |
| 性格変更     | エージェントの人格を変える |
| 噂を流す     | 情報操作          |
| 役職を変更   | バランス調整      |
| 記憶を変更   | ループ的操作      |
| 誰かを守る   | 推しキャラ保護    |
| 疑いを植え付ける | 社会操作      |

### 2.2 参加型（サブモード）

プレイヤーが1人のエージェントとして参加。他はLLMエージェント。

### 2.3 ハイブリッド型（理想）

| フェーズ       | プレイヤー        |
| -------------- | ----------------- |
| 通常           | 観戦              |
| 会議           | 観戦 or 参加      |
| 投票           | 参加可能          |
| 夜             | 観戦              |
| 特定イベント   | 介入              |

---

## 3. システム設計の基本方針

最重要原則：

> **「ゲームの真実・ルール」と「LLMの会話・人格」を完全に分離する」**

### 3.1 コンポーネント分担

| コンポーネント     | 役割                              |
| ------------------ | --------------------------------- |
| Game Engine (GM)   | ルール・役職・投票・勝敗・昼夜管理（**決定論的コード**） |
| Agent State        | 記憶・疑い・信頼・性格            |
| LLM                | 発言生成・推理・嘘・説得・感情    |
| Action System      | 投票・CO・夜行動                  |
| Logger             | ログ保存・リプレイ                |
| Player Interface   | 観戦・介入                        |

GMロジックはLLMの気まぐれに左右されない確実な進行管理として実装すること。

### 3.2 真実の所在

> **真実は必ずシステムが持つ。LLMには持たせない。**

| 種類     | 例                          |
| -------- | --------------------------- |
| Public   | CO・投票・死亡・発言        |
| Private  | 本当の役職・狼仲間・夜結果  |
| Personal | 疑い・信頼・推理            |

---

## 4. エージェント構造

各エージェントは以下の要素を持つ：

- 役職
- 性格（プロンプト）
- 記憶（長期・中期・短期）
- 信頼・疑いモデル
- 行動傾向
- 会話生成（LLM）
- 行動決定

---

## 5. プロンプト設計

各キャラクターへのプロンプトは **2つに分離**：

1. **性格プロンプト** — オープンにしてコミュニティがカスタマイズ可能
2. **役職プロンプト** — 村人・人狼・占い師等の役職ごとの行動指針

性格と役職を分離することで、同じ役職でも性格次第で戦略・発言が変わる。

---

## 6. 思考と発言の分離（灰ログ的演出）

- **思考**（腹の中）と**発言**（表の言葉）を別々に管理
- 思考過程の一部をあえて発言に滲ませる「灰ログ」的演出も検討
- 思考ログは観戦用・デバッグ用で表示モードを切り替え可能にする

## 8. 記憶設計

ログを全部渡すのはNG。**記憶は階層化する。**

| 記憶     | 内容                     |
| -------- | ------------------------ |
| 長期記憶 | 性格・他人の基本印象     |
| 中期記憶 | 今回のゲームの出来事     |
| 短期記憶 | 今日の議論・投票候補     |

キャラクターごとに状態ファイルを管理し、ゲームのフェーズ（昼夜）が進むたびに更新。更新されたファイルを次ターンのコンテキストとして渡すことでキャラクターの「経験」が蓄積される。

---

## 9. LLM出力の構造化

LLMの出力は文章だけでなく **JSON** にする。

```json
{
  "thought": "SQは昨日の票替えが不自然。おそらく人狼。",
  "speech": "今日はSQを疑っています。昨日の票替えが不自然でした。",
  "intent": {
    "vote_candidates": [
      {"target": "SQ", "score": 0.74},
      {"target": "Raqio", "score": 0.42}
    ],
    "co": null
  },
  "memory_update": ["SQの票替えを黒要素として維持"]
}
```

`thought` フィールドが灰ログ演出の基盤となる。最終投票はシステムが決定（疑い度・性格補正・ランダム・戦略）。

---

## 10. 行動はツール（MCP的設計）

行動は自然文ではなく **構造化アクション** として扱う。

| Action           | 内容 |
| ---------------- | ---- |
| vote(target)     | 投票 |
| co(role)         | CO   |
| accuse(target)   | 非難 |
| defend(target)   | 擁護 |
| ask(target)      | 質問 |
| guard(target)    | 護衛 |
| inspect(target)  | 占い |
| attack(target)   | 襲撃 |

フロー：LLM → 行動提案 → システム検証 → 実行

---

## 11. 人格パラメータ（育成要素）

従来ゲームのSTR/INTに相当するものが「人格」。

| パラメータ   | 意味     |
| ------------ | -------- |
| Logic        | 論理     |
| Emotion      | 感情     |
| Aggression   | 攻撃性   |
| Cooperation  | 協調性   |
| Suspicion    | 疑い深さ |
| Charisma     | 説得力   |
| Honesty      | 正直     |
| Deception    | 嘘能力   |
| Memory       | 記憶重視 |
| Intuition    | 直感     |

これをプロンプトに変換して人格を生成する。性格テンプレートはコミュニティ共有の仕組みも検討。

## 15. 将来の拡張方向

- **Web/モバイルアプリ化**: バックエンドはPythonのまま。FastAPI + WebSocketでリアルタイム観戦UIを提供。フロントはReact（Web）/ React Native（モバイル）
- **人間プレイヤーの参加**: ユーザー管理・認証をFastAPI側に追加し、人間が発言・投票に参加できるモード
- **状態管理のDB移行**: Web化・マルチゲーム並行時にJSONファイルからDBへ移行（MVPはファイルで可）
- **性格テンプレートのコミュニティ共有**: 性格プロンプトをオープンにしてユーザーがカスタマイズ・共有
- **エージェントごとのLLMモデル割り当て**: `AgentState` に `model` フィールドを持たせ、エージェントごとに異なるモデル（claude-opus / sonnet 等）をアサイン可能にする
- **Twitter/X的な会話構造**: 各発言にメッセージIDを振り、`reply_to` / `quote` フィールドで他発言を参照できるようにする。`public_log.jsonl` のエントリにIDを持たせるだけで対応可能。MVPの設計を壊さず将来追加できる
